## 2. 管理端周期绑定

**状态：** 已提供 **后台 API**（无可视化页面，可先 Postman / 后台二次开发）。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/adminapi/product/auction_session` | 周期列表 |
| POST | `/adminapi/product/auction_session` | 创建周期 `name,start_date,end_date` |
| POST | `/adminapi/product/auction_session/bind` | 绑定专场 `session_id`, `cate_ids: [id1,id2]` |

**示例（绑定两场限时拍）：**

```json
POST /adminapi/product/auction_session/bind
{
  "session_id": 1,
  "cate_ids": [10, 11]
}
```

绑定后请 **清除缓存键 `category`**。

## 4. 退款审核后台

**状态：** 已新增接口（替代仅写号牌表 `application` 的旧流程）。

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/adminapi/product/timed_deposit/pending` | `application=1` 待审列表 |
| POST | `/adminapi/product/timed_deposit/approve` | 审核退款 `uid`，调微信退 5000 并 `markRefunded` |
| POST | `/adminapi/product/timed_deposit/deduct` | 违约扣减 `uid`, `amount` |

用户端申请仍为：`POST` 号牌 `refund`（传 `cate_id`）。

旧接口 `POST /adminapi/product/refundOrder` 仍保留（按 cate 退号牌表），双场建议改用 `timed_deposit/approve`。

## 5. session 有效性校验

**状态：** 已实现。

- `StoreAuctionSessionServices::isValidSessionMode()`：周期存在，且绑定 **1～2** 个 `model=0` 专场。
- `resolveContext()`：仅 **有效 session** 才走模式 B，否则回退模式 A。
- `bindTimedCategories()`：绑定时校验专场类型、是否已绑其他周期、最多 2 场。
