# 竞价与代理出价改动说明

## 1. 背景与问题

小程序端「代理出价」跟价依赖服务端定时任务（`CrontabRunServices::automatic` / `synchronousAutomatic`）周期性执行 `StoreBiddingServices::automatic()` 与 `synchronousAutomatic()`。他人手动出价成功后，若未立刻触发上述逻辑，用户会感觉**代理跟价明显延迟**，直到下一次定时任务执行。

此外，部分代码路径在分类缓存未命中时使用了未定义的变量，存在**运行时致命错误**风险；代理逻辑若在手动出价已落库之后抛错，可能导致接口返回失败而数据已写入，**体验与排查**都不理想。

## 2. 涉及文件

| 文件 | 说明 |
|------|------|
| `app/services/product/product/StoreBiddingServices.php` | 竞价与代理出价核心服务，本次改动均在此文件 |

## 3. 改动摘要

### 3.1 出价成功后立即执行代理逻辑

- **普通拍** `addPrice()`：在手动出价成功（写库、清 `productList` 等完成之后）调用 `automatic()`，使代理在同一请求内尽快跟价，而不仅依赖定时任务。
- **同步拍** `synchronousAddPrice()`：在出价成功并清除 `productList1` 后调用 `synchronousAutomatic()`，行为与普通拍对齐。

上述调用外层增加 **try-catch**：代理或订阅消息等环节异常时，记录 `Log::error`，**仍返回「出价成功」**，避免手动出价已成功但接口返回 500。

### 3.2 分类缓存冷启动修复

- **`automatic()`**：补充实例化 `StoreCategoryDao`；在重建 `category` 缓存的循环中使用 `$StoreProductDao->getCount(...)`，避免未定义变量。
- **`synchronousAutomatic()`**：补充实例化 `StoreCategoryDao`，保证 `category1` 缓存未命中时 `getSynchronization` 可正常执行。

### 3.3 `addPrice()` 中分类列表 count 修复

- 当 `category` 缓存为空、需从数据库拉取并回填缓存时，将错误的 `$storeProductDao->getCount` 改为 **`$StoreProductDao->getCount`**（与当前方法内已实例化的商品 DAO 一致），避免分类缓存首次构建时报错。

### 3.4 日志

- 新增 `use think\facade\Log`。
- 捕获异常时写入：
  - `addPrice后代理出价失败: ...`
  - `synchronousAddPrice后代理出价失败: ...`

便于线上根据时间与堆栈排查订阅消息、数据库等次要环节问题。

## 4. 行为变化说明

| 项目 | 改前 | 改后 |
|------|------|------|
| 代理跟价触发时机 | 主要依赖定时任务（及原可选接口，若启用） | 每次**有效手动出价成功**后立即执行一轮全量代理扫描（与定时任务逻辑相同） |
| 定时任务 | 保留 | 保留，作为兜底 |
| 接口返回 | 与改前一致（成功/失败文案） | 与改前一致；代理内部异常不再拖垮整单 HTTP 结果 |
| 递归风险 | 无 | 无（`automatic` / `synchronousAutomatic` 不回调 `addPrice` / `synchronousAddPrice`） |

## 5. 对后续逻辑与运维的影响

1. **性能**：单次出价接口可能略慢，因为会多执行一轮与定时任务同级别的代理遍历（代理记录很多时需关注接口耗时与数据库压力）。若后续需要优化，可考虑按 `product_id` 只处理当前拍品的代理记录。
2. **并发**：商品行仍依赖 `StoreProductDao::automatic()` 的 `version` 乐观锁；与定时任务并发时，仍以最后一次成功更新为准，与改前同类并发场景一致。
3. **监控**：若代理失败被 try-catch 吞掉，用户仍看到出价成功，但代理可能未完全执行；需结合上述 **Log** 与业务监控判断是否需要告警或重试策略。
