# 号牌通用与双场拍卖需求说明（v0.8）

> **最终定稿请阅：** [号牌通用与双场拍卖-最终方案需求.md](./号牌通用与双场拍卖-最终方案需求.md)  
> 本文档保留讨论过程与版本记录；开发评审以 **最终方案** 为准。

本文档基于当前仓库实现梳理**限时拍**双场并行、号牌通用、保证金独立、保证金足额参拍的改造需求。

**关联文档：** [竞价与代理出价改动说明.md](./竞价与代理出价改动说明.md)

**v0.8（现网逻辑兼容，已确认）：**

- **现网：** **1 个限时拍专场 = 1 枚号牌**；该专场拍完后，**下一期**限时拍再办牌 → **生成新号牌**  
- **改造：** 仅当后台将 **最多 2 场** 限时拍绑定同一 `auction_session`（前后两天）时，**共用 1 枚号牌**  
- **未绑定 session 的旧专场 / 单专场活动** → **继续走现网 1 场 1 牌逻辑**，保证线上行为可用  

**v0.7（保证金金额，已确认）：**

- 限时拍保证金 **只有 5000 元** 这一档：**固定 5000**，无 1 万、无比例追加、无自选更高金额  
- 参拍足额 = **已缴满 5000 且未退款**（不是「≥5000 可无限追加」）

**v0.6 仍有效：** 周期结束未退可继续拍；退款后须再缴 **5000**。

**v0.5 仍有效：** 前后两天为一周期；1 牌 2 场；号牌与保证金独立。

## 一、业务目标（已确认）

| # | 目标 | 说明 |
|---|------|------|
| 0 | **前后两天为一周期** | **1 枚号牌 + 保证金 5000 元**；该周期内 **最多 2 场限时拍** 共用 |
| 1 | **双场并行** | 同一周期内，最多 **2 场限时拍**（`model=0`）可并行或先后处于预展/进行中 |
| 2 | **号牌通用（仅双场周期）** | 同一 `auction_session` 内 **办 1 次号牌**，两天内两场 **共用同一号牌号**；**下一期活动仍办新牌**（与现网一致） |
| 3 | **保证金仅 5000 元** | 限时拍保证金 **固定 5000 元**，无其他金额档位；周期内两场共用；**未退款可跨周期沿用** |
| 4 | **不足须缴纳** | 未缴满 **5000** 时须缴纳（**仅支持缴至 5000**，不支持超额缴纳） |
| 5 | **缴足后可继续拍** | 足额后正常出价/代理 **不扣减** 保证金本金 |
| 6 | **同时代理出价** | 两天内两场 Lot 上 **可同时** 设置代理并跟价 |
| 7 | **号牌与保证金独立** | 办卡、缴保证金分开；参拍需两者都满足 |
| 8 | **保证金仅限时拍** | 5000 体系只适用于限时拍 |

## 二、核心概念（必须先分清）

### 2.0 拍卖周期：前后两天（已确认）

**业务定义：** 运营安排的一场「双场限时拍」活动，时间跨度为 **连续 2 个自然日**（**前后两天**）。例如：

- **第 1 天**：限时拍专场 A（`cate_id` 之一）
- **第 2 天**：限时拍专场 B（`cate_id` 之二）

或两天内两场同时预展/进行中。**号牌**绑定本周期；**保证金**以用户账户为准，**未退则跨周期有效**。

```
auction_session（前后两天）
├── Day1 ── 限时拍专场 A
├── Day2 ── 限时拍专场 B
├── 号牌：本周期内 1 枚，两场均用（新周期须重新办卡）
└── 保证金：固定 5000 元；本周期两场共用；未退可延续至下周期继续拍
```

| 维度 | 规则 |
|------|------|
| 周期长度 | **2 个连续自然日**（`end_date - start_date = 1 天` 跨度，含首尾共 2 天） |
| 专场数量 | 周期内 **最多 2 个** `model=0` 专场，绑定同一 `session_id` |
| 号牌 | `user_id + session_id` **唯一**；**按周期**办理，新周期 **重新办卡** |
| 保证金 | **固定 5000 元**；`paid_total = 5000` 且 **未退款** 即可参拍 |
| 周期结束 | **不自动清零**；用户 **可申请退 5000**；**若未退，可继续拍** |
| 新周期 | **号牌**须重新办理；保证金 **仍为 5000 未退** → **无需再缴** |

**解析规则：** 前端传入任意 `cate_id` → 后端查其 `session_id` → 返回该周期号牌与保证金状态。

### 2.1 三个概念的关系

```
┌─────────────┐     ┌──────────────────────┐     ┌─────────────────┐
│   号牌      │     │  保证金（限时拍）      │     │  同步拍额度      │
│ 参拍身份标识 │     │ 固定押金 5000 元       │     │ remaining 不改   │
│ 4位数字展示  │     │ 仅 5000，无追加档位    │     │                 │
└─────────────┘     └──────────────────────┘     └─────────────────┘
      │                         │
      └──── 相互独立 ────────────┘
              均需满足才可参拍
```

| 概念 | 限时拍（本需求） | 同步拍（不在范围） |
|------|------------------|-------------------|
| 号牌 | **前后两天周期内** 1 枚，两场均用 | 现网仍按 `cate_id` 办卡（暂不改造） |
| 保证金 | **仅 5000 元**，缴满且未退即可拍；周期结束未退可继续拍 | 10 万 / 1:5 等（**不同体系**） |
| 出价额度 | **不设** `remaining` 池；**只校验保证金是否足额** | 仍用 `remaining` / `freeze` |

### 2.2 保证金行为（v0.7）

**金额规则（已确认）：**

| 规则 | 说明 |
|------|------|
| 唯一金额 | 限时拍保证金 **只有 5000 元**，无 1 万、无多档、无用户自选更高金额 |
| 支付接口 | `wxpay` 限时拍路径 **仅接受 `total_fee = 5000`**（与现网 `NotifyController` 5000 分支对齐） |
| 禁止项 | **无**「追加保证金」「比例保证金(1:5)」「专场 10w 保证金」等限时拍入口 |

| 场景 | 行为 |
|------|------|
| 未缴纳 | **不能**出价/代理；引导 **缴纳保证金 5000 元** |
| 已缴 &lt; 5000 | **不够**；须补缴至 **满 5000**（仍仅为凑满 5000，非追加档位） |
| 已缴 = 5000 且未退 | **足额**；两场可拍；正常出价 **不扣减** 这 5000 |
| 第二场 / 新周期 | **不再另缴 5000**（同一笔 5000 未退则沿用） |
| 违约扣除 | 若实缴被扣至 **&lt; 5000** → 须 **再缴满 5000** |
| 周期结束 | 保证金 **仍在（5000 未退）** → **可继续拍** |
| 退还 | 用户申请退款，退 **5000**；**退款后** 再参拍须 **重新缴 5000** |

**与同步拍区别：** 限时拍 **固定 5000 押金** + **无 `remaining` 额度池**；同步拍仍为 10w / 1:5 等 **另一套规则**。

**一句话：** 限时拍保证金 **只有 5000**；缴满且未退就能拍；**没有别的金额**。
### 2.3 参拍资格（限时拍目标态）

用户参与限时拍出价/代理，需同时满足：

1. **已办理号牌**（当前参拍 `session` 已办卡；新周期须在新 session 办卡）
2. **保证金有效**（**`paid_total = 5000`** 且 **未退款**；与周期是否结束无关）

**保证金不够时：** 前端展示「缴纳保证金」/「补缴保证金」；后端出价、设置代理接口 **拒绝** 并返回明确提示，引导完成支付。

**保证金足额后：** 可正常出价、代理；**不因单次出价再次要求缴纳**（除非违约扣除等导致再次不足）。

## 三、现状梳理（代码事实）

### 3.0 现网号牌逻辑（必须保持可用）

**业务现状（用户确认）：**

```
限时拍专场 A（cate_id = 101）  →  用户办号牌 → 得到号牌 1234
        ↓ 专场 A 结拍（status = 2）
限时拍专场 B（cate_id = 102，下一期）  →  用户再进专场 → 需再「办理号牌」→ 得到新号牌 5678
```

| 维度 | 现网行为 |
|------|----------|
| 绑定关系 | **1 个限时拍专场（`cate_id`）↔ 1 枚号牌** |
| 数据唯一 | `store_number_plate`：`user_id` + `cate_id` |
| 办卡入口 | `GET /numberPlate/add/:cate_id`，按专场判重（「不要重复办理号牌」） |
| 查牌 | `GET /numberPlate/details/:cate_id` |
| 号牌数字查重 | 同一 `cate_id` 内 4 位数不重复 |
| 下一期 | **新 `cate_id` = 新专场** → 用户 **重新办卡** → **新 `number_plate`** |
| 同时几场 | 列表 `getALlByIndex(1)`，**同一时间只驱动 1 场** |

**改造后与之关系：**

| 场景 | 号牌规则 |
|------|----------|
| 旧数据 / 未配置 `session_id` 的专场 | **完全按现网**：1 专场 1 牌，下一期新牌 |
| 新「双场周期」`auction_session`（前后两天，2 个 `cate_id`） | **周期内 1 牌 2 场**；**下一期新 session** → **仍须重新办卡、新号牌**（与现网「下一期新牌」一致） |

```
现网（单专场）                    目标（双场周期）
─────────────                    ─────────────────
cate_101 → 牌 A                  session_X
cate_102 → 牌 B（另一期）   →      ├── cate_101 → 牌 S（共用）
                                 └── cate_102 → 牌 S（共用）
session_Y（下一期）              session_Y → 牌 T（新办，与 S 无关）
```

### 3.1 限时拍 vs 同步拍

| 项目 | 限时拍 `model=0` | 同步拍 `model=1` |
|------|------------------|------------------|
| 列表 | `getcatlist` / `getALlByIndex(limit=1)` | `getSynchronization(limit=1)` |
| 前端 | `goods_cate1`、`goods_cate2` | `synchronouslist`、`synchronous_bidding` 等 |
| 出价接口 | `StoreBiddingController::auction` → `addPrice` | `synchronousAddPrice` |
| 代理任务 | `automatic()`，`agentlist model=0` | `synchronousAutomatic()`，`model=1` |
| 额度扣减 | `addPrice` **基本不扣** `remaining` | `synchronousAddPrice`、代理 **扣/回** `remaining` |
| 代理设置 | `StoreAgentPriceServices::saveData` 在 `model==1` 才扣额度 | 同步拍扣 `remaining` |

**结论：** 限时拍后端对额度的依赖较弱；主要问题在 **数据模型把号牌与保证金揉在同一张表**，以及 **专场数量、号牌按 cate_id 隔离**。

### 3.2 号牌与保证金在现网的耦合（需改造）

现网 `store_number_plate` **一行同时承载**：

| 字段 | 现网含义 | 问题 |
|------|----------|------|
| `number_plate` | 号牌数字 | 应按周期通用，现按 `cate_id` 重复 |
| `state` | UI 上「号牌是否激活/保证金是否已缴」混用 | 无法表达「有牌未缴」「已缴未办牌」 |
| `paystatus` | 保证金支付状态 | 写在号牌表上，与号牌办理耦合 |
| `remaining` / `freeze` | 竞拍额度（同步拍为主） | 限时拍目标为 **不限制**，应停止写入/校验 |
| 唯一键 | `user_id` + `cate_id` | 现网 **一场一牌**（正确）；双场周期需 **额外** `session_id` 维度 |

支付回调 `NotifyController`：5000 元成功时更新 **号牌表** `state=1, paystatus=1`，进一步加深耦合。

前端 `goods_cate1.vue` 文案：`号牌{{plate}}, 请缴纳保证金` —— 同一条 UI 把两件事绑在一起。

### 3.3 双场并行现状

- `getALlByIndex($limit=1)`：同时只加载 **1 个**限时拍专场
- 竞价、WebSocket、订阅消息等多处使用 `category[0]`，默认全场唯一

### 3.4 代理出价现状

- `store_agent_price` 按 `product_id` + `uid` 存储，含 `cate_id`、`model`
- 限时拍代理（`model=0`）设置时 **不扣** `remaining`
- 跟价 `automatic()` 全量扫描 `model=0` 代理记录，**已支持多商品**；瓶颈在专场列表与号牌按 `cate_id` 隔离

### 3.5 关键文件

| 模块 | 文件 |
|------|------|
| 号牌 | `StoreNumberPlateServices.php`、`StoreNumberPlateController.php` |
| 保证金 | `PayController.php`、`NotifyController.php`、`StoreMarginDao.php` |
| 限时拍出价/代理 | `StoreBiddingServices.php`（`addPrice`、`automatic`） |
| 同步拍出价/代理 | `StoreBiddingServices.php`（`synchronousAddPrice`、`synchronousAutomatic`）— **本次不改** |
| 代理 CRUD | `StoreAgentPriceServices.php` |
| 专场 | `StoreCategoryServices.php`、`StoreCategoryDao.php` |
| 小程序限时拍 | `goods_cate1.vue`、`goods_cate2.vue` |

## 四、目标态行为

### 4.1 双场并行（仅限时拍）

- 同一拍卖周期内，最多 **2 个** `model=0` 专场同时处于预展或进行中
- `getcatlist` / 相关缓存返回 **最多 2 条** 专场记录
- 所有依赖 `category[0]` 的逻辑改为按 **`cate_id` 或专场列表** 定位，禁止隐式假设只有一场

### 4.2 号牌（独立业务）

**双场周期（有 `session_id`）：**

- 用户在 **本 session** 内 **办理 1 次**，生成 **1 个** 4 位号牌
- 进入该 session 下 **任一场** `cate_id`，查询到 **同一** `number_plate`
- 同 session 第二场 **不再**「办理号牌」

**单专场 / 无 session（兼容现网）：**

- 行为与 **§3.0 现网** 一致：`user_id` + `cate_id` 一条牌
- 专场结拍后，**下一期新 `cate_id`** 用户再办卡 → **新号牌**

**下一期双场周期：**

- 新 `auction_session` → 用户 **重新办理号牌**（新 4 位数），**不沿用** 上一期 session 的号牌

号牌办理 **不触发** 保证金支付；号牌表 **不存** 保证金字段（限时拍新路径）。

### 4.3 保证金（独立业务，仅限时拍）

- **固定 5000 元**，两场限时拍 **共用** 同一用户保证金账户（非多档、非超额）
- **不足 5000** → 须缴纳/补缴至 **满 5000**；**不接受** 单笔支付金额 &gt; 5000 的限时拍保证金订单
- **足额（5000 且未退）后**：正常出价、代理 **不扣减** 本金；**可继续拍**
- 保证金记录 **独立于号牌表**；缴纳 **不自动办卡**
- **周期结束不作废**：**5000 未退** → 仍可参拍（含下一周期）
- **退款**：退 **5000**；退款后须 **重新缴 5000**

### 4.4 限时拍与 `remaining`（区别于同步拍）

- 限时拍 **不使用** 同步拍式 `remaining` 可出价额度池（无 1:5 比例）
- 参拍校验：**`paid_total === 5000` 且未退** + **当前 session 已办卡**
- `StoreBiddingServices::addPrice`、`automatic`（`model=0`）：校验保证金足额；**不读写** `remaining`
- 前端 `goods_cate1/2`：**不展示**「剩余额度」；不足时展示 **缴纳/补缴保证金**
- **同步拍** `remaining` 逻辑 **不改**
### 4.5 同时代理出价（限时拍）

- 用户可在专场 A、专场 B 的多个 Lot 同时设置 `store_agent_price`（`model=0`）
- 设置代理前校验：**号牌已办 + 保证金足额**；不足则引导缴纳
- `automatic()` 对两场均生效；**无 `remaining` 不足失败**（仅保证金门槛）

## 五、数据与接口方案（推荐）

### 5.1 拍卖周期 `auction_session`（前后两天）

| 字段 | 说明 |
|------|------|
| `id` | 批次 ID |
| `start_date` | 周期 **第 1 天**（含当日 00:00:00） |
| `end_date` | 周期 **第 2 天**（含当日 23:59:59）；与 `start_date` 相差 **1 天**，共 **2 个自然日** |
| `cate_ids` | 关联 **最多 2 个**限时拍 `store_category.id`（分属前后两天或周期内并行） |
| `status` | 批次状态（未开始 / 进行中 / 已结束） |

**约束：**

- 一个 `session` = **前后两天** = **1 号牌体系 + 1 保证金账户 + 最多 2 场限时拍**
- 后台创建两场限时拍时，**必须**绑定同一 `session_id`，并配置落在该 2 日窗口内
- `cate_id` 不在当前 session 日期范围内 → 不按该 session 的号牌/保证金处理

### 5.2 号牌表（双模式兼容）

**模式 A — 现网（保留）：** `user_id` + `cate_id`（`session_id` 为空或专场未纳入双场周期）

| 字段 | 说明 |
|------|------|
| `user_id` | 用户 |
| `cate_id` | 限时拍专场 |
| `number_plate` | 4 位，**专场内**唯一 |
| 行为 | 与线上一致：一场一牌；下一期新 `cate_id` 再办 → 新号 |

**模式 B — 双场周期（新增）：** `user_id` + `session_id`（同一 session 下最多 2 个 `cate_id` 共用）

| 字段 | 说明 |
|------|------|
| `user_id` | 用户 |
| `session_id` | `auction_session.id` |
| `number_plate` | 4 位，**session 内**唯一 |
| `created_at` | 办理时间 |

**接口解析（`numberPlate/add`、`details`）：**

```text
cate_id → 查 store_category.session_id
  若 session_id 有效且 session 含 2 场 → 模式 B（按 session 判重、查牌）
  否则 → 模式 A（按 cate_id，与现网相同）
```

缓存：模式 A 保留 `Plates_{uid}{cate_id}`；模式 B 使用 `Plate_{uid}_{session_id}`。

**历史数据：** 已有 `user_id + cate_id` 记录 **只读保留**，不做合并；新双场活动走模式 B。

### 5.3 保证金表（限时拍 · 用户级为主）

保证金以 **用户维度** 为主账户（可关联 `session_id` 记录缴款来源，但 **有效性不按周期过期**）：

| 字段 | 说明 |
|------|------|
| `uid` | 用户 |
| `paid_total` | 限时拍保证金实缴，**足额为 5000**（退款后为 0） |
| `paystatus` | 0 未缴满 / 1 **已缴满 5000**（可继续拍）/ 2 **已退**（须再缴 5000） |
| `session_id` | 可选，记录缴款关联周期 |
| `out_trade_no` | 微信订单号 |

**参拍判断：**

```text
deposit_sufficient = (paid_total == 5000) AND (paystatus == 1)
```

**支付创建：** 限时拍 `wxpay` 仅允许 `price/total_fee = 5000`。

周期内两场共用；**5000 未退则周期结束后仍可拍**。

支付回调 **只更新保证金记录**；**不回写号牌表**。

参拍资格接口建议返回：

```json
{
  "number_plate": "1234",
  "has_plate": true,
  "paid_total": 5000,
  "deposit_sufficient": true,
  "need_pay": 0
}
```

`need_pay`：未缴满时为 `5000 - paid_total`（最大补差至 5000，**无超额缴纳**）。
### 5.4 兼容与同步拍

- 历史 `store_number_plate`（按 `cate_id`）只读保留；新周期走 session 模型
- 同步拍继续读旧表或独立逻辑；**不在本次迁移中改动** `remaining`/`freeze` 行为

## 八、验收标准（v0.8）

**A. 现网兼容（单专场 / 无 session）**

1. 仅 1 个限时拍专场时，与改造前一致：**1 场 1 牌**。  
2. 用户在专场 A 办卡后，专场 A 结拍；进入 **下一期** 专场 B（新 `cate_id`）→ 可再办卡，得到 **新号牌**。  
3. `numberPlate/add/:cate_id`、`details/:cate_id` 对 **无 session** 专场行为不变。

**B. 双场周期（有 session）**

4. 一个 `auction_session`（前后 2 天）绑定 **最多 2 场**；周期内办牌 **1 次**，两场 **同号**。  
5. **下一期新 session** → **重新办卡、新号牌**（不沿用上期号码）。  
6. 保证金 **仅 5000**；未缴满/已退款不能拍；**5000 未退** 可继续拍（含跨 session）。  
7. 两场 **同时代理** 跟价正常。

**C. 通用**

8. 限时拍 **无** 10w、1:5、追加保证金入口。  
9. **同步拍** 行为不变。
